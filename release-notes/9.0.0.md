# 9.0.0

Two things, and they are both about your page not being the only shape a page
comes in. A theme can now apply to part of a page instead of all of it. And
zen-ui works in Arabic, Hebrew and Persian — which, honestly, it did not before,
in ways a screenshot would not have shown you.

## What breaks

**A `:root` override of a `--zen-*` token may start winning where it used to
lose.**

Theme tokens used to be declared at `:root[data-theme="dark"]`. They are now
`[data-theme="dark"]` — one specificity step lower, which is what lets a theme
apply to a `<div>`. If you wrote your overrides like this:

```css
/* loaded AFTER zen-ui's stylesheet */
:root {
  --zen-color-primary: #7c3aed;
}
```

that rule used to lose to zen-ui's theme block and silently do nothing. It now
ties on specificity, so source order decides, and yours wins. **If some of your
overrides were dead, they are about to come alive.** That is the direction you
wanted, but it can change what your app looks like without you touching it.

Nothing to do if your overrides were already working. If your palette shifts,
the rules that changed are the ones you had given up on.

**In a right-to-left page, several components now behave differently — because
they were wrong.** Arrow keys, a carousel's direction of travel, which side a
submenu opens on. If you built around the old behaviour, that workaround is now
the bug. Details below.

## A theme for part of the page

`applyTheme()` sets the theme for the whole document. Sometimes you want one
panel to disagree — a dark code viewer in a light app, a preview that shows what
the other theme looks like, an embedded widget that must keep its own palette
whatever the host page does.

```jsx
<Theme name="dark">
  <Card>Everything in here is dark.</Card>
</Theme>
```

It nests, without you keeping track of anything:

```jsx
<Theme name="dark">
  <Card>dark</Card>
  <Theme name="zen-theme">
    <Card>zen-theme, not dark</Card>
  </Theme>
</Theme>
```

No JavaScript runs. CSS custom properties inherit, and the nearest themed
ancestor wins — which is why nesting needs no bookkeeping and no level counting.

`<Theme transparent>` renders the wrapper as `display: contents` for when it sits
directly inside a grid or flex container and must not become a box of its own.

**One thing it does not do yet.** A `Dialog`, `Popover`, `Sheet`, `Tooltip` or
`DropdownMenu` opened from inside a `<Theme>` renders through a portal into
`<body>`, which is outside the themed element — so the overlay comes back with
the document's theme. Its own demo page says so plainly rather than leaving you
to find out.

Available in all four bindings; the web-components one is `<zen-theme name="dark">`.

## Right-to-left actually works now

The honest summary of the previous state: an RTL page **looked** right and
**behaved** left. Layout mirrored correctly, because CSS does that on its own.
Everything decided in JavaScript did not.

**Arrow keys now follow the reading direction.** In Arabic or Hebrew, "next" is
to the *left*. Sixty places in the library disagreed — the carousel, the rating,
the NPS and Likert scales, one-time-password boxes, the tree, the colour
palette, the object page's section anchors, the slider. Every one of them
treated `ArrowRight` as "forward", which is correct in English and backwards
everywhere else.

**The carousel did not move at all.** Separate bug, same area, and the arrow-key
fix alone would not have caught it: in a right-to-left container the browser
counts scroll position *downwards from zero into negative numbers*, so the
carousel was scrolling the wrong way and stopping dead at the start.

**Menus, tabs, sliders and selects need one line from you.** The primitives
underneath zen-ui keep direction in their own JavaScript, and it does not read
`dir` off your page. Render this once, near the root:

```jsx
import { DirectionProvider } from "@algorisys/zen-ui-react";

<DirectionProvider>
  <App />
</DirectionProvider>
```

With no props it follows `<html dir>` and keeps following it — so a language
switch at runtime works without a reload. Pass `dir="rtl"` to state it outright.
It renders no element, so it cannot affect your layout.

*Solid takes an optional `locale` here that React does not.* Not an
inconsistency we chose: Kobalte works out direction *from* a locale rather than
accepting a direction, so on that binding the two are tied together. Pass
`locale` if you care about how dates and numbers are formatted.

**Text alignment follows the reading direction.** Table headers used to stay
left-aligned in an RTL page while their own column data moved right, so headers
did not line up with the columns they named. Sixty-nine places, all of them
meaning "align the way this reads" rather than "align left".

**The time picker deliberately does not flip.** Clock notation is
left-to-right in every locale — hours first — so its segments stay in order
while the page around them mirrors.

## Also

- zen-ui's own components work out direction from where they sit in the page, so
  an RTL panel inside an LTR page is correct with no configuration.
- `Theme` and `DirectionProvider` both ship in all four bindings.
