# 9.4.0

Three additions to `TreeTable`. Nothing you already wrote changes.

If you shipped 9.3.0's `TreeTable`, everything here is opt-in behind a flag —
your existing tables render exactly as before.

## Trees too big to send whole

`TreeTable` used to need the entire tree up front. Now it can fetch a subtree
the first time you open it.

```jsx
<TreeTable
  data={roots}
  columns={columns}
  getRowId={(row) => row.id}
  hasChildren={(row) => row.childCount > 0}
  loadChildren={(row) => fetch(`/api/nodes/${row.id}/children`).then((r) => r.json())}
  onLoadChildrenError={(err) => toast.error(String(err))}
/>
```

**`hasChildren` is the one that makes it work.** A row that has not loaded yet
looks exactly like a leaf, so without it there is no chevron and no way to open
the row that would have triggered the load. Say which rows can be opened and the
chevron appears; opening one calls `loadChildren`.

While the fetch is in flight the chevron becomes a spinner and the control
reports `aria-busy`, so the row does not jump to a placeholder that may be
replaced 40ms later. Results are cached per row, so collapsing and re-opening
does not re-fetch.

This needs `getRowId` (or an `id` on your rows). The cache keys on that id — an
index-based key would move the moment anything above it is sorted or filtered,
and you would silently re-fetch rows you already had.

`onLoadChildrenError` is optional; without it a rejected promise is re-thrown
rather than swallowed.

## Pagination that does not orphan a subtree

```jsx
<TreeTable
  data={data}
  columns={columns}
  enablePagination
  pageSize={2}
  pageSizeOptions={[2, 5, 10]}
/>
```

**`pageSize` counts top-level rows, not rendered rows.** A page carries each
root's whole subtree, so the number of rows on screen changes as you expand
things while the number of roots stays put.

That is deliberate, and it is why this took a version to arrive. The obvious
implementation pages the flattened list of visible rows — and that cuts through
a subtree, putting half of someone's children on page 2 underneath no parent at
all. Paging the roots is the only version that is true of a hierarchy.

The bar reads `Page 1 of 2 · 3 top-level rows` for the same reason: saying
"3 rows" would contradict what you can count on screen the moment anything is
open.

## Windowing, for a tree you expand all of

```jsx
<TreeTable
  data={bigTree}
  columns={columns}
  enableVirtualization
  maxBodyHeight={360}
/>
```

Only visible rows were ever in the DOM, so a large tree sitting collapsed always
cost nothing. The case that hurt was expanding all of a big one — around 22,600
open rows put 162,000 nodes on the page and took about a second to mount. With
this on, only the rows near the viewport render: measured at 1,240 open rows,
16–25 of them are in the DOM instead of 1,240.

It needs `maxBodyHeight`, because without a bounded scroller there is no window
to speak of. Leave it out and it warns in the console rather than quietly doing
nothing. `rowEstimatedHeight` (default 44) only governs the scrollbar before a
real row has been measured.

Windowed tables also carry `aria-rowcount` and `aria-rowindex`, since the DOM no
longer holds every row and a screen reader would otherwise be told the table is
25 rows long.

## Faster expanding in vanilla and web-components

Opening or closing a node used to rebuild every row on the page to change one.
It now touches only the affected subtree. At 1,110 visible rows a toggle went
from about 49ms to 3–8ms; at 22,600 rows, from about 924ms to 88–138ms.

No API change — the same clicks, less waiting. React and Solid were never
affected; their renderers already did this.

## All four bindings

Everything above is in React, Solid, vanilla and web-components, and all four
were driven through the same browser checks. On `<zen-tree-table>`,
`hasChildren`, `loadChildren` and `pageSizeOptions` are set as JS properties
rather than attributes — there is no attribute form of "call this and await a
promise".
