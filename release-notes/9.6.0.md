# 9.6.0

One new component, in all four bindings. Nothing you already wrote changes.

## PlanningCalendar — who is busy, and when

One row per resource, one axis of time, appointments as blocks on it.

```jsx
const rows = [
  {
    id: "r1",
    title: "Rhea Iyer",
    subtitle: "Field engineer",
    appointments: [
      {
        id: "a2",
        start: new Date(2026, 6, 21, 10),
        end: new Date(2026, 6, 21, 11, 30),
        title: "Install",
        subtitle: "Bay 4",
        state: "success",
      },
    ],
  },
];

<PlanningCalendar rows={rows} />
```

This is the component for questions you answer by reading *across* rows — who is
free on Thursday, which room is double-booked, when the van is back. Every
resource shares one time axis, so the comparison is a vertical glance rather
than three tabs.

**A month is one axis of 31 columns, not a 6×7 page.** Wrapping a month into
weeks would give every resource six separate rows, which destroys exactly the
comparison you came for. If you want a month page with one date per cell, that
is `Calendar` — a different question, a different component.

Three views: `day` (hours), `week` (days, Monday first), `month` (days). The
toolbar moves one view at a time and `Today` comes back. Give it `views` to
offer fewer, or `view` + `onViewChange` to drive it from your own toolbar. Same
for `date` / `onDateChange`.

### It shows the plan; it does not edit it

There is no drag-to-move, no drag-to-create and no resize handle. Those need a
conflict policy, an undo story and a permission model that belong to your
domain, and a component that half-implements them is worse than one that
plainly does not.

```jsx
<PlanningCalendar
  rows={rows}
  onAppointmentClick={(appointment, row) => openEditor(appointment, row)}
/>
```

You get the appointment *and* its row, because the row is half the answer to
"what did I just click".

### What it decides for you

- **Overlaps stack.** Two appointments at the same time get their own lanes, so
  a double-booking is visible rather than hidden behind whichever drew last. It
  uses exactly as many lanes as the busiest moment needs.
- **Back-to-back is not an overlap.** 10:00–11:00 followed by 11:00–12:00 share
  a lane. Otherwise an ordinary day turns into a staircase.
- **Appointments that run past the edge are cut, not squared off.** A block
  continuing into tomorrow loses its border and corner on that side, so it reads
  as continuing rather than ending there.
- **A zero-length appointment is still visible and clickable.** A milestone at
  14:00 gets a real sliver; 0% wide is indistinguishable from data that failed
  to load.
- **The "now" line only appears when now is actually in view.** A line pinned to
  the left edge of next week reads as "it is Monday morning", which is worse
  than no line.
- **Below about 3% of the width, a block shows no text.** A 90-minute meeting is
  0.9% of a week: a label there is not "clipped", it is an empty bordered pill
  that reads as a failed render. The block becomes an occupancy mark instead,
  and its name and time stay in the tooltip and the accessible name. Switch to
  the day view to read them.
- **Every block is a real button.** They are the only things in the grid worth
  reaching by keyboard.

### Times are yours

`start` and `end` are your `Date` objects, in local time, unconverted. A
calendar that converted would have to be told which zone to convert *to*, and
getting that wrong moves every appointment by hours with nothing looking broken.
If your data is UTC, convert before you pass it.

### In the other bindings

```js
PlanningCalendar({ rows, defaultView: "week" }).el;
```

```html
<zen-planning-calendar rows='[
  { "id": "r1", "title": "Rhea Iyer", "appointments": [
    { "id": "a2", "start": "2026-07-21T10:00", "end": "2026-07-21T11:30",
      "title": "Install", "state": "success" }
  ] }
]'></zen-planning-calendar>
```

JSON has no date type, so the element revives ISO strings into `Date`s for you.
Setting `el.rows` with real `Date`s works too, and is the usual route once you
have data rather than markup.

```js
el.addEventListener("zen-appointment-click", (e) => {
  const [appointment, row] = e.detail;
  openEditor(appointment, row);
});
```
